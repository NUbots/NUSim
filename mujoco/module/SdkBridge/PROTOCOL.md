## 1. IDL struct layouts

All 9 structs below come from `include/booster/idl/b1/*.h` + `include/booster/idl/rpc/*.h`,
which are themselves fastddsgen-generated headers (`GEN_API_VER == 2`, i.e. generated by a
Fast-DDS 2.x-era fastddsgen). **The private member declaration order in these headers is the
CDR serialization order** — fastddsgen always emits struct members, and therefore
serialize()/deserialize() field-by-field calls, in IDL declaration order, which is
mirrored 1:1 into the generated class's private member list. This holds even though the
*getter/setter method* order in the header can differ slightly from repo to repo; we
always read the **private member list**, not the method list, as authoritative.

Note: the headers we can read declare classes directly as e.g.
`class booster_interface::msg::LowState : public eprosima::fastdds::dds::TopicDataType`
(data + serialization fused, no `dds_` namespace visible in the public header). But
`strings` on the compiled `.a` shows the *registered* DDS type name is
`booster_interface::msg::dds_::LowState_` (see §2) — extra `dds_::` module and a
trailing `_` on the struct name that aren't in the visible class name. The public
headers are therefore a cosmetically-renamed/consolidated copy of the true underlying
IDL, which itself nests every struct three levels deep
(`module pkg { module msg { module dds_ { struct Name_ { ... }; }; }; };`, ROS2's
`rosidl_adapter` convention). §5 records exactly how we reproduce that nesting and the
generator invocation empirically confirmed to reproduce the registered name byte-for-byte.

## 2. Registered DDS type names (verified byte-identical between SDK versions)

`strings lib/x86_64/libbooster_robotics_sdk.a | grep 'dds_::'` on **both** the pinned
tarball (`324946e7`) and the newer local checkout (`7fb7287`, used as our contract-test
client) yields the same names for every type we use:

```
booster_interface::msg::dds_::LowState_
booster_interface::msg::dds_::MotorState_
booster_interface::msg::dds_::ImuState_
booster_interface::msg::dds_::Odometer_
booster_interface::msg::dds_::FallDownState_
booster_interface::msg::dds_::BatteryState_
booster_interface::msg::dds_::ButtonEventMsg_
booster_interface::msg::dds_::LowCmd_
booster_interface::msg::dds_::MotorCmd_
booster_msgs::msg::dds_::RpcReqMsg_
booster_msgs::msg::dds_::RpcRespMsg_
```

Also present (not used by us): `booster_interface::msg::dds_::RemoteControllerState_`,
`RobotStatesMsg_`/`RobotStatesMsg_` (older tarball only — renamed away in the newer
checkout), `FallDownRecoveryState_`, various `audio`/`sensor_msgs`/`geometry_msgs`
types, confirming the whole SDK follows the ROS2 `pkg::msg::dds_::Type_` convention
consistently, including for genuinely-ROS2-standard types like
`builtin_interfaces::msg::dds_::Time_` — strong independent evidence this is literally
ROS2's `rosidl_adapter`-generated `.idl` module nesting (`module pkg { module msg {
module dds_ { struct Type_ { ... }; }; }; };`), not an SDK-specific invention.

Our `mujoco/idl/**/*.idl` files reproduce this exact 3-level module nesting, and plain
fastddsgen (see §5 for why **not** `-typeros2`) then registers each type under the
identical fully-qualified scoped name — verified directly against the generated
`*PubSubTypes.cxx` (`setName(...)` string literal) after each generation, see
`mujoco/idl/regenerate.sh` output and the M4 acceptance test.

**Empirical finding on `-typeros2`** (the task's suggested starting point was
`module booster_interface { module msg { struct <Name>_ {...}; }; };` — 2-level module,
trailing-underscore struct name — generated with `-typeros2`): tested directly against
`/opt/k1sim-deps/bin/fastddsgen` (v3.2.1) with a throwaway `Odometer_` struct in three
configurations:
1. 2-level module + `-typeros2` → registers as `booster_interface::msg::dds_::Odometer__`
   (note the **double** trailing underscore — `-typeros2` appends `_` to a name that
   already ends in `_`). **Wrong.**
2. 3-level module (explicit `dds_` nesting, as above) + `-typeros2` → registers as
   `booster_interface::msg::dds_::dds_::Odometer__` (doubles the namespace **and** the
   underscore — `-typeros2` is not idempotent against pre-nested input). **Wrong.**
3. 3-level module (explicit `dds_` nesting) + **no** `-typeros2` → registers as
   `booster_interface::msg::dds_::Odometer_`. **Exact match.**

Conclusion: `-typeros2` itself performs the `dds_`-wrapping + trailing-`_` transform
(useful when starting from plain ROS2 `.msg`-style IDL that doesn't already have the
nesting); since we author the nesting ourselves (to match the SDK's exact structure,
including cases like `FallDownStateType_` where the SDK uses a genuine IDL `enum`
rather than ROS2 `.msg`-style integer constants — not something `-typeros2` would
produce from a `.msg` mirror), passing `-typeros2` on top double-applies the transform.
`mujoco/idl/regenerate.sh` therefore does **not** pass `-typeros2`.

## 3. RPC JSON schemas

Source: `include/booster/robot/rpc/{request_header,response_header}.hpp` +
`include/booster/robot/b1/b1_loco_api.hpp` + `b1_api_const.hpp` (pinned SDK).

`RpcReqMsg_.header` JSON (`booster::robot::RequestHeader`):
```json
{"api_id": <int64>, "expect_response": <bool, default true>}
```
`RpcRespMsg_.header` JSON (`booster::robot::ResponseHeader`):
```json
{"status": <int64>}
```
(`0` = success; SdkBridge returns `config/dds.yaml: unknown_api_status` (default `0`)
for unrecognised `api_id`, with a WARN log.)

Per-call `body` JSON (all confirmed against `b1_loco_api.hpp`'s `...Parameter`/`...Response`
classes' `FromJson`/`ToJson`):

| api_id | name | request body | response body |
|---|---|---|---|
| 2000 | `kChangeMode` | `{"mode": <int>}` (`RobotMode`) | — |
| 2001 | `kMove` | `{"vx": <float>, "vy": <float>, "vyaw": <float>}` | — |
| 2004 | `kRotateHead` | `{"pitch": <float>, "yaw": <float>}` | — |
| 2007 | `kLieDown` | `""` (empty) | — |
| 2008 | `kGetUp` | `""` (empty) | — |
| 2017 | `kGetMode` | `""` (empty) | `{"mode": <int>}` |
| 2025 | `kGetUpWithMode` | `{"mode": <int>}` | — |
| 2038 | `kVisualKick` | `{"start": <bool>, "version": <int>}` (`VisualKickVersion`: `kV1=0` default, `kV2=1`) | — |

`RobotMode`: `kUnknown=-1, kDamping=0, kPrepare=1, kWalking=2, kCustom=3, kSoccer=4`
(`include/booster/robot/common/robot_shared.hpp`) — numerically identical to
`shared/k1/BoosterApi.hpp::RobotMode` (frozen contract, unchanged).

All other `LocoApiId` values (`b1_api_const.hpp`/`b1_loco_api.hpp` defines ~40 more —
hand gestures, dances, WBC gait, custom-trained-trajectory playback, etc.) are outside
NUbots_K1's call set and are **unknown ids** for us: SdkBridge logs a WARN and replies
`{"status": <dds.yaml: unknown_api_status>}` (default `0`, i.e. "accept silently" —
matches `B1LocoClient::SendApiRequest`'s 1000 ms blocking wait, so any unimplemented
call still returns promptly instead of timing out).

`B1LocoClient::SendApiRequest` blocks up to `timeout_ms` (default 1000 ms) waiting for
a `RpcRespMsg_` with matching `uuid` on `rt/LocoApiTopicResp`; NUbots' `HardwareIO` uses
the default. SdkBridge **always replies synchronously inside the DDS listener callback**
so replies are sub-millisecond, never dependent on the physics thread.

NUbots calls `ChangeMode(kSoccer)` immediately at startup and `ChangeMode(kPrepare)` at
shutdown (verified in NUbots_K1 HEAD, out of scope for this workstream but load-bearing
context for why `kSoccer=4`/`kPrepare=1` must dispatch correctly from t=0).

## 4. QoS

`include/booster/robot/channel/channel_factory.hpp`'s `ChannelFactory::CreateSendChannel
/ CreateRecvChannel` both default `reliable = false` (best-effort) unless the caller
opts in; `DdsFactoryModel::SetWriter/SetReader` only upgrades
`reliability().kind` to `RELIABLE_RELIABILITY_QOS` when `reliable == true`, otherwise
leaves the class's own `writer_qos_`/`reader_qos_` defaults (Fast-DDS defaults:
`BEST_EFFORT`, `KEEP_LAST` history depth 1, `VOLATILE` durability). **The actual
per-channel `reliable` argument values are baked into the compiled `.a`**
(`RpcClient`/`B1LocoClient`/`low_state` subscriber sources are not shipped, only
headers) — but the pinned SDK's own runtime logging revealed them during the contract
test: `ChannelSubscriber::InitChannel: setting reliability: 0, queue_capacity: 64` for
every state subscription (low_state/odometer_state/battery_state → **BEST_EFFORT,
KEEP_LAST(64)**) and `setting reliability: 1, queue_capacity: 1024` for the RPC
channels (→ **RELIABLE, KEEP_LAST(1024)**). Both pair legally with our choices below
(RELIABLE writer ≥ BEST_EFFORT reader; RELIABLE ↔ RELIABLE for RPC).

Per the lead's explicit spec (de-risks RPC correctness over squeezing out best-effort
UDP savings — reliability never hurts a loopback/`--network host` transport, and the
SDK's own defaults would silently degrade to best-effort which is a *safe* superset
to widen, not narrow), `module::SdkBridge` uses:
- **State writers** (`low_state`, `odometer_state`, `fall_down`, `battery_state`,
  `button_event`): `RELIABLE` + `VOLATILE` durability + `KEEP_LAST(5)` history.
- **RPC request reader** (`rt/LocoApiTopicReq`): `RELIABLE` + `KEEP_LAST(10)` history
  (bursts of calls at startup — e.g. NUbots issuing `ChangeMode` immediately — must not
  drop).
- **RPC response writer** (`rt/LocoApiTopicResp`): `RELIABLE` + `VOLATILE` +
  `KEEP_LAST(5)` (matches state writers; replies are small and infrequent).
- Domain: `0` (hardcoded in NUbots' `HardwareIO`, matches `config/dds.yaml: domain`).
- Transport: default UDPv4 + SHM (both enabled) unless `config/dds.yaml: udp_only: true`
  or env `K1_DDS_UDP_ONLY=1`, which strips SHM from the participant's
  `transport_descriptors` (workaround for docker `--ipc host` not always being
  available, e.g. rootless docker or restricted CI).

Empirically verified in the M4/M5 contract test (`test/contract/test_sdk_roundtrip.py`
+ its C++ helper client, built against **both** the older local SDK checkout at
`/home/nubots/Workspace/booster/booster_robotics_sdk` (commit `7fb7287`) **and** the
exact pinned tarball (`324946e7`)): RELIABLE writer QoS interops cleanly with the
SDK's BEST_EFFORT state readers (a RELIABLE writer + BEST_EFFORT reader pairing is
legal in DDS — the reader simply doesn't get retransmits it doesn't ask for), and
the RPC round trip completes in well under 1 ms against the client's 1000 ms
timeout in all cases — see the Results section of `test/contract/README.md` for
actual numbers.

**Operational gotcha discovered while validating**: Fast-DDS SHM segments live in
the host-shared `/dev/shm` (`--ipc host`). If the sim container is ever run **as
root** (i.e. without `docker/k1sim.sh`'s `--user $(id -u)`), it leaves root-owned
`fastrtps_port*` ring buffers behind that a later uid-1000 participant mapping to
the same port can neither write to nor delete (`/dev/shm` sticky bit) — host→sim
RPC then silently times out while some sim→host topics still flow, a very confusing
partial failure. Always run via `docker/k1sim.sh` (which passes `--user`), and if
mixed-uid segments ever accumulate, remove the stale `fastrtps_*` /
`fast_datasharing_*` files or set `udp_only: true`.

## 5. CDR / fastddsgen generation notes

The SDK's headers assert `GEN_API_VER == 2` (i.e. were generated by a Fast-DDS
2.x-era fastddsgen against classic CDR). The toolchain available to us
(`/opt/k1sim-deps/bin/fastddsgen`, baked into `k1sim:latest`) is **3.2.1**, whose
*default* output targets XCDR2 (`-cdr v2`, `-default_extensibility appendable`) —
which is wire-**incompatible** with the classic-CDR/`final`-extensibility structs the
pinned SDK generates (different (D)HEADER framing). We therefore invoke fastddsgen with
`-cdr v1 -default_extensibility final` (see `mujoco/idl/regenerate.sh`) to reproduce
classic CDR, matching what a 2.x-era generator (and ROS2's classic `rmw_fastrtps_cpp`
profile, which these `dds_::Type_` names impersonate) would have produced. This choice
was validated empirically by the contract test actually deserializing our published
`LowState_`/`Odometer_`/etc. samples with the real SDK's compiled (2.x-generated)
reader code — see `test/contract/README.md`.
